# planosh — 결정적 실행 계획을 위한 제안

planosh는 프레임워크가 아니다. **"AI 코딩 팀이 plan.sh를 써야 한다면, 그 plan.sh는 어떤 원칙을 따라야 하는가"**에 대한 제안이다.

이 문서는 포맷을 정의하지 않는다. 원칙을 정의한다. 실제 plan.sh가 어떻게 생겨야 하는지는, 여러 프로젝트에 써보면서 패턴을 찾아가야 할 문제다. 현재 우리가 가진 증거는 autonomous.sh 하나뿐이고, 하나의 사례로 프레임워크를 확정하는 것은 조기 추상화다.

## 문제 정의

AI 코딩 도구가 팀 전체에 보급되면서, 이제 개발자뿐 아니라 기획자, 디자이너도 코드를 만들 수 있게 되었다. 하지만 결정적인 실행 계획 없이 각자가 AI에게 "알아서 해줘"를 시키면, 팀 전체의 코드베이스가 비결정적으로 변한다.

현장에서 실제로 일어나는 일:

> **개발자**: "그건 claude로 해봐야 알 것 같아요..ㅠㅠ"
>
> — 계획이 없으니 실행 전까지 결과를 예측할 수 없다. "해봐야 안다"가 개발 프로세스가 되어버렸다.

> **기획자**: "저 추가하고 싶은 기능이 있어서 한번 개발하고 PR 올려봤어요. 리뷰해주세용."
> → 파일 변경 40개, +4,567 -12,300
>
> — 의도는 좋지만, 결정적인 계획 없이 AI에게 통째로 맡기면 이런 PR이 나온다. 만약 이 기능의 실행 계획이 먼저 리뷰되었고, 그 계획이 결정적이었다면? 코드를 한 줄도 안 봐도 됐다. 계획이 곧 결과를 보장하니까.

> **3개월 후**: "이 인증 코드, 왜 이 구조로 만들었지?"
> → AI 세션은 이미 사라졌고, 커밋 메시지는 "feat: add auth"뿐이다. 왜 JWT를 골랐는지, 왜 파일을 이렇게 나눴는지, 아무도 모른다.
>
> — 계획이 리뷰되었다면, 모든 기술 결정과 그 이유가 `.plan/`에 남아있다.

> **기획자가 명세를 보고 AI로 기능을 만들었다. 동작한다.** 일주일 후, 같은 명세로 작은 수정을 시켰더니, AI가 전혀 다른 구조로 만들어버렸다. 기획자는 코드를 읽을 수 없으니 뭐가 달라졌는지 모른다.
>
> — 명세는 있었다. 하지만 같은 명세에서 같은 결과가 나오지 않았다. **결정적이지 않은 계획은 계획이 아니다.**

네 사례의 공통점: **결정적인 실행 계획의 부재**. 팀의 모든 구성원이 AI로 개발할 수 있게 된 시대에, "무엇을 어떤 순서로 어떻게 만들지"가 결정되지 않은 채 실행하는 것은 재앙이다.

스펙 기반 개발(SDD) 도구들은 이 문제를 "좋은 마크다운 명세를 쓰면 AI가 알아서 좋게 구현한다"로 풀려고 했다. 하지만 마크다운 명세가 결정적으로 코드가 되지는 않는다. "리뷰된 명세"와 "동작하는 구현" 사이의 간극에는 여전히 해석이 필요하고, 바로 거기서 비결정성이 들어온다.

더 나쁜 건, **AI는 명세의 여백을 마주칠 때마다 '더 많이 만드는 쪽'으로 달린다**는 것이다. 로그인 하나 요청했는데 MFA, 이메일 인증, rate limiting, 추상 레이어 3개가 같이 나온다. 40 파일 변경, +4,567 -12,300짜리 PR이 SDD 도구의 산출물로 흔하게 태어난다. "이건 만들지 마세요"를 명세에 써도 소용없다 — 명세 전체가 프로즈라서, 같은 입력에 다른 출력이 나오는 구조 자체가 바뀌지 않는다. SDD 도구들은 이 갭을 그럴듯한 UI 뒤에 숨겼을 뿐이고, 그 안에서 AI의 과잉 구현은 멈추지 않는다.

근본적인 문제: **명세는 읽기 위한 문서이지, 실행하기 위한 지시가 아니다.** AI 에이전트가 명세를 읽을 때마다 재해석이 일어난다. 다른 세션, 다른 해석, 다른 결과.

## 첫 시도: plan.sh

이 문제에 대한 첫 번째 답은 단순했다: **계획을 실행 가능한 셸 스크립트로 만들자.** `claude -p "프롬프트"` 호출과 `verify` 검증을 bash 스크립트에 직접 넣으면, 계획과 실행 사이의 간극이 사라진다.

```bash
# Step 2: Google OAuth 로그인
CURRENT_STEP=2; step 2 "Google OAuth 로그인"
run_claude "
Google OAuth 로그인을 구현하세요.
## 만들 것
- User/Account/Session 모델 (prisma)
- NextAuth Google Provider + Prisma Adapter
## 하지 않을 것
- 이메일/비밀번호 가입
"
verify "빌드 성공" "npm run build"
checkpoint "feat: Google OAuth login"
```

이것은 동작했다. 하지만 plan.sh 자체에 답이 덜 된 질문들이 있었다:

1. **가독성** — 계획을 리뷰하는 사람이 bash를 읽어야 한다면, "코드 대신 계획을 리뷰하자"는 전제와 모순된다.
2. **순차 실행의 한계** — Step 1, 2, 3을 순서대로 실행하고 실패하면 멈춘다. 재시도도, rollback도, 실패에서 배우는 것도 없다.
3. **백엔드 종속** — `claude -p`가 하드코딩되어 있다.
4. **git 복잡성** — 브랜치, scope 체크, commit, worktree를 전부 bash로 작성해야 한다.
5. **상태 관리** — 실패 컨텍스트, 진행 이력, regression 정보를 어디에 어떻게 저장할 것인가.

이 질문들에 대한 답은 실전에서 나왔다.

## 배운 것: autonomous.sh

plan.sh로 실제 프로젝트를 돌려봤을 때, 자연스럽게 나온 구조는 순차 실행이 아니었다. **수렴 루프**였다.

```bash
# autonomous.sh — 16시간 무인 실행에 성공한 스크립트의 구조

while [ "$SESSION" -lt "$MAX_SESSIONS" ]; do
  # ① 다음 할 일 발견 (verify가 NEXT:를 반환)
  NEXT=$(verify --skip-tsc | grep '^NEXT:')

  # ② stuck 감지 (같은 작업 N번 연속 실패 → 중단)
  if [ "$FEATURE" = "$LAST_FEATURE" ]; then
    STUCK_COUNT=$((STUCK_COUNT + 1))
    [ "$STUCK_COUNT" -ge "$THRESHOLD" ] && break
  fi

  # ③ 프롬프트 조립 (태스크 + 진행상태 + 실패 컨텍스트)
  PROMPT="... ${FEATURE_JSON} ... ${REGRESSION_CONTEXT} ..."

  # ④ AI 실행
  claude -p "$PROMPT" --permission-mode bypassPermissions

  # ⑤ regression 체크 → 실패 시 rollback + 컨텍스트 저장 + 재시도
  if verify --regression | grep -q 'REGRESSIONS FOUND'; then
    git reset --hard "$BEFORE_SHA"
    REGRESSION_CONTEXT="$DETAILS"  # 다음 시도에 주입
    continue
  fi

  # ⑥ 성공 → commit + 배포
  checkpoint "$FEATURE"
  ota_publish "$FEATURE"
done
```

이 스크립트를 한 번 실행하고 퇴근했다. 16시간 후 모든 feature가 완료되어 있었다. 중간에 키보드를 만진 사람은 아무도 없었다.

이 경험이 가르쳐준 것:

1. **순차 실행(step 1→2→3)은 수렴 루프의 특수 경우다.** 할 일 목록이 정적이고, 실패 시 멈추도록 설정한 수렴 루프.
2. **실패 컨텍스트를 다음 시도에 주입하면 AI가 같은 실수를 반복하지 않는다.** "regression이 있었으니 조심하라"는 한 줄이 다음 시도의 성공률을 끌어올렸다.
3. **stuck 감지는 필수다.** 무한 루프를 막는 건 단순한 카운터 하나였다.
4. **진행 상태를 파일에 기록해두면 AI 세션 사이에 맥락이 유지된다.** `claude-progress.txt` 하나로 에이전틱 건망증을 상당 부분 막을 수 있었다.

이 네 가지는 autonomous.sh에서 검증된 패턴이다. 하지만 이것이 **모든 plan.sh가 따라야 할 유일한 형태**인지는 아직 모른다. 사례가 하나뿐이기 때문이다.

그럼에도 이 경험에서 plan.sh가 **어떤 원칙**을 따라야 하는지에 대한 방향은 나왔다.

## 원칙: 결정성

planosh에는 원칙이 하나뿐이다. **같은 플랜이 같은 코드로 수렴해야 한다.** 이 한 가지가 이뤄지면 나머지는 전부 따라 나온다 — 해석의 여지가 없어서 과잉 구현이 일어나지 않고, 결과가 예측 가능해서 무인 실행이 도박이 아니게 되며, 플랜이 결과를 대리하기 때문에 팀 전원이 코드 대신 플랜을 리뷰할 수 있다.

### 왜 결정성이 유일한 메트릭인가

AI가 생성한 코드의 품질을 평가하는 방법들:

- **코드 품질 평가 (AI reviewer)**: "좋은 코드인가" — 주관적이고 컨텍스트 의존적. AI evaluator의 판정도 비결정적. 비결정적 evaluator로 비결정적 생성기를 개선하는 루프는 수렴하지 않는다.
- **시각적 테스트**: "의도한 UI와 일치하는가" — AI의 시각적 판단 자체가 매번 다르다. 범용 eval로 사용 불가.
- **유닛 테스트/타입 체크**: 객관적이지만, "통과했다"가 "의도한 구현이다"를 보장하지 않는다.
- **결정성 (병렬 실행 diff)**: 같은 입력에 같은 출력이 나오는가. 주관이 아니라 사실. diff가 있으면 비결정적이고, 없으면 결정적이다. 모든 step, 모든 프로젝트에 동일하게 적용된다.

**결정성만이 AI 코드 생성에서 자동화된 개선 루프를 돌릴 수 있는 유일한 객관적 메트릭이다.** 나머지 품질 기준은 부차적이거나 결정성 위에서 도는 파생물이다.

### 결정성은 측정된다

결정성은 주장이 아니라 **측정할 수 있는 숫자**다. 같은 `plan.sh`를 격리된 환경에서 N번 병렬 실행하고 결과 diff를 본다. 발산 지점이 보이면, 그 지점이 아직 결정적이지 않은 곳이다. 봉쇄하고 다시 측정한다. 이 루프가 **calibration**이며, 구체적 동작은 뒤의 *교정(Calibration)* 섹션에서 다룬다.

"스크린샷이 예쁜가", "아키텍처가 우아한가" 같은 판정을 AI에게 맡기지 않는다. "결과가 수렴했는가"만 본다. 이 단호함이 루프를 수렴 가능하게 만든다.

## 결정성을 달성하는 실천 규칙

원칙만으로는 코드가 짜이지 않는다. autonomous.sh와 초기 `plan.sh` 실험에서 **결정성을 실전에서 지키는 데 필요했던 세 가지 규칙**이 드러났다. 각 규칙은 결정성을 다른 방식으로 보호한다:

- **규칙 1 (가독성)** — 결정성의 결과인 "플랜 리뷰가 코드 리뷰를 대체한다"를 현실화하는 전제 조건.
- **규칙 2 (수렴 루프)** — 실패 앞에서도 결정성을 유지하기 위한 실행 모델.
- **규칙 3 (WHAT/HOW 분리)** — 해의 공간을 구조적으로 좁혀 결정성을 원천적으로 확보하는 메커니즘.

세 규칙은 모두 결정성을 위한 도구다. 독립된 최우선 축으로 착각하면 원칙이 흩어진다.

### 규칙 1: 사람이 읽기 쉬워야 한다

계획이 결정적이면, 사람은 생성된 코드를 읽을 필요가 없다. +4,567줄의 PR이 올라와도 계획이 리뷰되었다면 승인할 수 있다. **계획 리뷰가 코드 리뷰를 대체한다.** 이 대체가 성립하려면 한 가지 조건이 추가로 필요하다 — 사람이 **플랜 자체를 읽을 수 있어야** 한다.

- 기획자가 "이 step에서 무엇을 만드는가"를 한눈에 이해할 수 있어야 한다
- 디자이너가 "이 기능의 범위가 어디까지인가"를 판단할 수 있어야 한다
- 주니어 개발자가 "여기에 검증이 부족하다"고 지적할 수 있어야 한다

plan.sh가 bash 스크립트라면 이 조건을 충족하기 어렵다. boilerplate(`step()`, `run_claude()`, `verify()`, `checkpoint()`)가 내용보다 많고, 프롬프트 텍스트가 heredoc에 묻힌다. **리뷰어가 bash를 읽을 줄 알아야 계획을 리뷰할 수 있는 것은 모순이다.**

이 규칙에서 파생되는 실천:

- boilerplate를 최소화한다. 프롬프트, 검증, 커밋 메시지가 스크립트의 주인공이어야 한다.
- 프롬프트는 자연어로 읽힌다. 기술 결정을 프롬프트에 반복하지 않는다.
- diff가 깔끔해야 한다. 프롬프트 한 줄을 수정했을 때 git diff에 그 한 줄만 나와야 한다.

구체적인 형식(bash인가 YAML인가 다른 무엇인가)은 이 규칙을 가장 잘 충족하는 것을 고르면 된다. 지금은 어느 형식이 정답인지 모른다.

### 규칙 2: 수렴 루프로 실패를 소화한다

결정성이 높은 플랜도 실행 도중 실패할 수 있다 — 환경 차이, transient error, AI 세션의 미묘한 흔들림. 단순 순차 실행(step 1→2→3)은 이런 실패 한 번에 깨진다. autonomous.sh가 증명한 것: 실전에서 동작하는 것은 **실패에서 배우는 루프**다.

수렴 루프의 최소 요소:

- 실패를 감지한다 (verify + regression check)
- 실패하면 rollback한다 (git reset to snapshot)
- 실패 컨텍스트를 다음 시도의 프롬프트에 주입한다
- stuck이면 멈춘다 (같은 작업 N회 연속 실패)

순차 실행은 이 루프의 특수 경우다. 할 일 목록이 정적이고, 실패 시 멈추도록 설정하면 순차 실행이 된다. 따라서 **수렴 루프를 기본으로 설계하고, 순차 실행은 그 위의 설정**으로 취급한다. 수렴 루프가 있어야 같은 플랜을 돌렸을 때 결과가 일관되게 수렴한다 — 없으면 한 번의 실패가 비결정성의 문을 열어둔다.

### 규칙 3: WHAT과 HOW를 분리한다

plan.sh의 프롬프트에는 **이 step에서 무엇을 만드는가**만 들어간다.

- "만들 것" — 이 step의 구체적 산출물
- "하지 않을 것" — 이 step의 범위 외 항목

코딩 컨벤션, 아키텍처 규칙, 금지 패턴은 **하네스**에 들어간다. 이것들은 한 번 합의하면 모든 step에 적용되는 것이고, 매 step마다 리뷰할 필요가 없다.

이 분리는 결정성을 두 층위에서 보호한다:

1. **해의 공간 압축**: 프롬프트가 WHAT을 제약하고 하네스가 HOW를 제약하면, 같은 입력이 해석될 수 있는 공간 자체가 좁아진다. 좁은 공간에서는 AI 세션 간 편차가 줄어든다.
2. **우선순위 보호**: 시스템 프롬프트(`--append-system-prompt`)는 유저 프롬프트보다 우선순위가 높다. 컨벤션을 하네스에 넣으면 AI가 "잊을" 가능성이 낮다.

부수적으로 가독성에도 기여한다 — 리뷰어가 매 step마다 같은 컨벤션 규칙을 건너뛰지 않아도 된다. 프롬프트에 WHAT만 있으면 step마다 정말 달라지는 것만 보인다.

## 결정성 모델: 3계층 제약

결정성 원칙과 WHAT/HOW 분리 규칙을 합친 구조 — 결정성이 목표, 분리가 그 목표를 달성하는 구조적 메커니즘이다:

```
┌─────────────────────────────────────────────────────┐
│ Layer 1: 하네스 (시스템 프롬프트)                      │
│ HOW — 코딩 컨벤션, 아키텍처 규칙, 금지 패턴            │
│ "어떻게 만들지"를 제약                                │
├─────────────────────────────────────────────────────┤
│ Layer 2: 프롬프트                                    │
│ WHAT — 만들 것, 하지 않을 것, 선행 조건               │
│ "무엇을 만들지"를 제약                                │
├─────────────────────────────────────────────────────┤
│ Layer 3: 검증 (verify + regression)                 │
│ CHECK — 빌드, 파일 존재, 테스트 통과, regression      │
│ 결과가 제약에 부합하는지 사후 확인                     │
└─────────────────────────────────────────────────────┘
```

해의 공간(solution space) 압축:

```
  마크다운 명세만          프롬프트만             프롬프트 + 하네스
  ┌─────────────┐       ┌─────────────┐       ┌─────────────┐
  │ ░░░░░░░░░░░ │       │             │       │             │
  │ ░░░░░░░░░░░ │  →    │   ░░░░░░░   │  →    │    ░░░      │
  │ ░░░░░░░░░░░ │       │   ░░░░░░░   │       │             │
  │ ░░░░░░░░░░░ │       │             │       │             │
  └─────────────┘       └─────────────┘       └─────────────┘
  해석 자유도: 높음       WHAT 제약됨            WHAT + HOW 제약됨
```

프롬프트가 **무엇을** 만들지 제약한다면, 하네스는 **어떻게** 만들지 제약한다. 두 축이 교차하면 해의 공간이 좁아진다. verify는 그 공간 안에 결과가 들어갔는지 사후 확인한다.

## 하네스 구조

```
.plan/
└── {plan-name}/
    ├── harness-global.md           ← 모든 step에 적용
    ├── harness-step-1.md           ← step별 맥락 제약
    ├── harness-step-2.md
    └── ...
```

**글로벌 하네스** — 프로젝트 전체에 일관되게 적용:

```markdown
# 프로젝트 컨벤션

## 기술 결정

- D-001: Next.js 14 App Router + TypeScript strict mode
- D-002: Prisma ORM, PostgreSQL
- D-003: Google OAuth only (NextAuth.js)

## 코딩 규칙

- 컴포넌트: src/components/{feature}/{ComponentName}.tsx
- 'use client' 최소화 — 서버 컴포넌트 우선
- CSS: Tailwind만 사용, 인라인 style 금지

## 절대 금지

- any 타입 사용
- 하드코딩된 시크릿
- 이 step의 범위 밖 파일 수정
```

**Step별 하네스** — 해당 step의 맥락에 맞는 추가 제약:

```markdown
# Step 2 하네스: Google OAuth 로그인

## 현재 프로젝트 상태

Step 1 완료 후 존재하는 파일:

- prisma/schema.prisma (빈 스키마)
- src/app/layout.tsx, src/app/page.tsx

## 이 step의 아키텍처 제약

- NextAuth 설정: src/lib/auth.ts (단일 파일)
- 세션 전략: JWT (database 세션 아님)

## 생성할 파일 목록 (이 목록 외 파일 생성 금지)

- prisma/schema.prisma
- src/lib/auth.ts
- src/app/api/auth/[...nextauth]/route.ts
- src/middleware.ts
```

Step별 하네스의 "현재 프로젝트 상태" 섹션은 에이전틱 건망증을 방지한다. AI가 이전 step의 산출물을 추측하지 않고 하네스에서 읽는다.

"생성할 파일 목록" 화이트리스트는 범위 초과를 막는다. AI가 요청하지 않은 파일을 만들면 verify 후 커밋 단계에서 경고가 뜬다.

## 교정 (Calibration)

하네스가 실제로 해의 공간을 좁혔는지 측정하는 방법이 필요하다. 이것이 calibration이다.

```
같은 plan.sh를 격리된 환경에서 N번 병렬 실행
  ↓
결과물 간의 diff 분석 (구조/패턴/네이밍/범위)
  ↓
발산 지점 발견 → 사용자에게 결정 요청
  ↓
결정을 하네스에 규칙으로 추가 → 재측정
```

발산 자체가 문제가 아니다. 발산은 **하네스가 아직 제약하지 못한 지점을 가리키는 신호**다. 3번 실행해서 세션 전략이 JWT/JWT/database로 갈렸다면, 어느 쪽이 맞는지는 사람이 결정할 일이다. calibration의 역할은 "여기서 갈라진다"를 발견하고, 사용자에게 결정을 받아 하네스에 기록하여 더 이상 갈라지지 않게 만드는 것이다.

발산 유형:

- **구조 발산** — 파일/디렉토리 구조가 다름 → "생성할 파일 목록" 화이트리스트로 봉쇄
- **패턴 발산** — 같은 파일인데 구현 패턴이 다름 → 명시적 아키텍처 제약 추가
- **네이밍 발산** — 변수명/함수명/컴포넌트명이 다름 → 네이밍 규칙 추가
- **범위 발산** — 요청하지 않은 기능이 추가됨 → "절대 금지" 항목 추가

calibration은 "더 나은 코드"를 찾는 루프가 아니라 "발산 지점"을 찾는 루프다. 결정성이 유일한 eval이라는 원칙의 직접적인 구현이다.

## planosh가 하지 않는 것

planosh는 **PRD가 이미 존재한다고 가정**한다. PRD 작성, 아이디어 정리, 요구사항 도출은 planosh의 범위가 아니다.

```
아이디어 → PRD 작성     |     PRD → 결정적 실행
━━━━━━━━━━━━━━━━━━━━━  |  ━━━━━━━━━━━━━━━━━━━━━
planosh 범위 밖          |  planosh 범위
```

planosh는 PRD를 받아서 결정적인 실행 계획으로 변환하는 것에 집중한다. PRD의 품질이 plan.sh의 품질을 결정하므로, 좋은 PRD를 먼저 준비하는 것이 중요하다.

## 아직 답하지 못한 질문

원칙은 있지만 구체적인 답은 아직 없는 것들. 이것들은 더 많은 실전 사례가 쌓여야 답할 수 있다.

1. **포맷**: plan.sh가 bash여야 하는가? YAML이나 다른 형식이 가독성에 더 도움이 되는가? 포맷을 미리 정하기보다 여러 형식을 시도해보고 고르는 것이 맞아 보인다.

2. **수렴 루프의 표현**: 수렴 루프를 bash로 매번 손으로 쓸 것인가, 공통 패턴을 추출해서 라이브러리로 만들 것인가? 지금은 autonomous.sh의 while 루프를 복붙해서 쓰는 방식이 현실적이다.

3. **백엔드 추상화**: `claude -p`에서 `codex`나 다른 도구로 갈아탈 수 있게 하려면 어떤 레이어가 필요한가? 지금 답하기엔 다른 도구들의 실제 동작을 충분히 써보지 않았다.

4. **상태 관리**: `.plan-state`, regression context, progress 파일, calibration history를 어떻게 구조화할 것인가? 프로젝트마다 다른 것 같은데 공통점이 있는지 아직 모른다.

5. **계획 생성의 수준**: `/planosh` 스킬이 PRD에서 plan.sh를 얼마나 자동으로 생성할 수 있는가? 사람이 수동 편집해야 하는 부분은 어디인가?

6. **스킬과 프레임워크의 경계**: 스킬이 할 일과 (언젠가 만들) 프레임워크가 할 일의 경계는 어디인가?

이 질문들에 답하려면 5-10개의 plan.sh 실전 사례가 필요하다.

## 다음 단계

1. **여러 프로젝트에 plan.sh 적용** — 다양한 도메인(웹앱, CLI, 마이그레이션, 컴파일러)에서 plan.sh를 작성해본다.
2. **패턴 수집** — 작성된 plan.sh들에서 반복되는 구조를 식별한다. 결정성 원칙과 세 실천 규칙을 지키는 가장 가독성 높은 형식이 무엇인지 본다.
3. **발산 측정** — calibration을 돌려서 각 패턴이 실제로 결정적인지 확인한다.
4. **공통 추출** — 사례가 충분히 쌓이면 공통 패턴을 추출해서 프레임워크 설계를 재검토한다.

지금은 **"이렇게 생겨야 한다"가 아니라 "이런 원칙을 지켜야 한다"** 수준에서 멈춘다. 원칙은 분명하지만 구현은 열려 있다.

## 설계 원칙

1. **planosh는 제안이지 프레임워크가 아니다.** 정답을 정의하지 않는다. 원칙을 정의한다. 실제 패턴은 커뮤니티가 찾는다.
2. **증명된 것만 넣는다.** autonomous.sh에서 실전으로 검증된 패턴만 원칙으로 올린다. 이론적으로 좋아 보이지만 써보지 않은 것은 "아직 답하지 못한 질문"에 넣는다.
3. **가독성은 형식 결정의 기준이다.** planfile의 형식(bash/YAML/기타)을 고를 때 "사람이 읽기 쉬운가"를 기준으로 판단한다. 계획이 코드 리뷰를 대체하려면 계획 자체가 코드보다 읽기 쉬워야 한다.
4. **조기 추상화를 경계한다.** 사례 하나로 프레임워크를 만들지 않는다. 5-10개가 쌓이면 그 때 공통점을 뽑는다.
